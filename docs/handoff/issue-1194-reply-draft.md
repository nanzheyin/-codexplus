## 回复正文

> **@xuweibing233** 这个问题现在有完整的解决方案了。我做了一个 PR 来解决纯文本模型（DeepSeek-V4、GLM-5.2 等）遇到图片和 reasoning 的报错问题，已经提了 #1405 的同系列 PR。

### 现在可以做什么

**方案一：静默丢弃图片（strip）**

在 Codex++ 的供应商配置 → 模型列表中，给 DeepSeek-V4-pro 勾选「只支持文本」。之后 Codex++ 会在转发前自动丢掉所有 `input_image` 块，不会再报 `unknown variant image_url`。

> 注意：这个功能仅对 **Chat Completions 协议**的中转生效。如果你用的是 Responses 协议（直连），则不会经过代理，图片原样发给上游。

**方案二：让视觉模型帮你"看图说话"（VL 中转）**

如果你不想丢图片，而是想让模型"看懂"图片内容：

1. 在 Codex++ 设置 → 视觉模型中转（VL）→ 启用
2. 配置一个支持图片的视觉模型（如 Qwen-VL-Plus、GPT-5.5 等）
3. 给 DeepSeek-V4-pro 勾选「只支持文本」

之后 Codex++ 会：
- 拦截你的请求中所有的图片
- 调用 VL 模型描述图片内容（带你的提问文字一起发给 VL，让描述更精准）
- 把文字描述替换图片，再交给 DeepSeek-V4-pro

这样纯文本模型也能"看懂"图片了，而且之前被图片污染的会话也能恢复（图片被替换为文字描述）。

### 和 #1405 的区别

我和 @kanchengw 分别独立做了 VL 中转的实现，两个方案殊途同归，都解决了同一个问题。他的 PR (#1405) 功能更全面（含缓存、历史轮次分析等），我的更精简（只处理当前请求，适合快速合并）。最终合并哪个由维护者决定，但对用户来说效果是一样的——纯文本模型不再报错，图片可以被可视化模型描述替代。

### 总结

| 你的问题 | 现在的状态 |
|---------|-----------|
| DeepSeek-V4-pro 上传图片报 JSON 解析异常 | ✅ 勾选「只支持文本」即可解决 |
| 想保留图片信息、让模型看懂图 | ✅ 配置 VL 视觉模型中转即可 |
| 之前被图片污染的会话 | ✅ 开启 VL 后，代理会自动替换历史图片为文字描述 |
